Chapter 0

Setting Up Your Environment

Before the first line of React, the boring-but-essential five minutes of setup. Do this once and forget it.

0.1 What You Need Installed

React itself is a JavaScript library, but in 2026 nobody writes React by dropping a <script> tag into an HTML file. We use a build tool that handles JSX, TypeScript, hot reloading, and bundling. The build tool we will use throughout this book is Vite — it is fast, minimal, and the de facto choice for new React projects, including every React Three Fiber starter you will ever encounter.

You need exactly three things on your machine:

  1. Node.js (LTS, version 20 or newer) — the JavaScript runtime that powers the build tool. Download from nodejs.org and accept the defaults. Installing Node also installs npm, the package manager.
  2. A code editor — Visual Studio Code is the universal standard. It is free and has perfect React/TypeScript support out of the box.
  3. A modern browser — Chrome, Edge, Firefox, or Brave. You will live in the developer tools panel, so pick one you like opening.

To verify Node installed correctly, open a terminal and type:

node --version
npm --version

If both print version numbers, you are done with installation. If either errors, restart your terminal — the PATH variable updates only on new sessions.

0.2 Creating Your First React + TypeScript Project

Vite ships a project generator. From any folder on your computer, run:

npm create vite@latest aurelia-residences -- --template react-ts
cd aurelia-residences
npm install
npm run dev

That is the entire setup. Line by line:

  • npm create vite@latest aurelia-residences tells npm to run the Vite scaffolder and name the new folder aurelia-residences.
  • -- --template react-ts tells Vite to use the React + TypeScript template. (If you prefer plain JS, use react instead — but this book defaults to TypeScript and you should follow along.)
  • cd aurelia-residences moves your terminal into the new folder.
  • npm install downloads every dependency listed in package.json into a folder called node_modules.
  • npm run dev starts the Vite development server, usually at http://localhost:5173. Open that URL in your browser and you will see the Vite + React starter page.
🔑 Key Idea

Vite watches your files. The moment you save a change in your editor, the browser reloads and shows the new state. This feedback loop is the single most important thing about modern frontend development — make sure it works before moving on.

0.3 What's Inside the Project

Open the folder in VS Code. You will see a structure like this:

aurelia-residences/
├── node_modules/        (libraries — never edit, never commit)
├── public/              (static files served as-is)
├── src/                 (your code lives here)
│   ├── App.tsx          (the root React component)
│   ├── main.tsx         (the entry point that mounts App to the page)
│   ├── index.css        (global styles)
│   └── assets/
├── index.html           (the single HTML file Vite serves)
├── package.json         (project manifest and scripts)
├── tsconfig.json        (TypeScript configuration)
└── vite.config.ts       (Vite configuration)

For the next fifteen chapters you will spend essentially all of your time inside the src/ folder. Open src/App.tsx — that file is your first React component, even if it does not look like much yet.

0.4 The Mental Model of a React File

A React file is a JavaScript (or TypeScript) file that exports a function. That function returns something that looks suspiciously like HTML. That "HTML-inside-JavaScript" is called JSX, and the rest of the book will unpack it. For now, just know: a React component is a function that returns a description of UI.

⚠ Common Mistake

Editing files inside node_modules. Never do this. Treat that folder as read-only. If you need to change the behavior of a library, you wrap it, not edit it.

0.5 Stopping and Restarting the Server

To stop the dev server, focus the terminal and press Ctrl+C (Windows/Linux) or Cmd+C (macOS). To start it again, run npm run dev. You can leave the server running in the background while you work — Vite will catch up to your edits automatically.

0.6 Optional: Useful VS Code Extensions

You can install React in five minutes with zero extensions, but a few are worth having:

  • ES7+ React/Redux/React-Native snippets — type rfc and get a complete component scaffold.
  • Prettier — auto-formats your code on save so you stop arguing with yourself about indentation.
  • Error Lens — shows TypeScript errors inline next to the offending code instead of hidden in a panel.

Chapter 1

What React Is — and Why It Exists

Every tool solves a problem. Understanding React starts with understanding that problem.

1.1 The Problem React Solves

Imagine you are building a web page that shows the price of a house. Somewhere in your JavaScript, you hold a variable: let price = 450000. Somewhere in your HTML, there is a span that displays that price. Now imagine the price changes. You have a problem, and the problem is keeping these two things in sync.

Before React, developers solved this by finding the span in the DOM and updating it by hand: document.querySelector('.price').textContent = price. This works for one value. It collapses under its own weight once you have twenty values, nested components, and a dozen places in your code that can update them. You end up with a nest of "when X changes, remember to also update Y, Z, and W" bugs. Whole careers have been lost to this nest.

React's entire reason for existing is to make the nest go away. Instead of you manually updating the DOM whenever data changes, you describe what the UI should look like for a given piece of data, and React takes care of the updates.

🔑 Key Idea

React is not a framework for styling or routing or anything else. React is a machine for keeping the screen in sync with your data, automatically, so you never have to manually find and update DOM elements again.

1.2 The Declarative Idea in One Sentence

Describe what the UI should look like for a given state, and let React figure out how to get there.

This sentence is going to feel thin until you have written a few React components. But it is the whole idea. Hold on to it.

🧠 Mental Model

Think of React as a smart foreman. You hand the foreman a blueprint that says "for a price of $450,000, the page should look like this." The foreman walks the site, figures out exactly what needs to change to match the blueprint, and makes only those changes. You never touch a hammer.

1.3 React as a Runtime

React is a JavaScript library that runs in the browser. When your React code runs, React builds an in-memory representation of what the screen should look like. When your data changes, React builds a new representation, compares it to the old one, and applies only the differences to the real DOM. This process is called reconciliation, and it is what makes React feel almost magical when you first use it.

1.4 The Virtual DOM, Without the Hype

You have probably heard the phrase "Virtual DOM" said with reverence. It just means: React keeps a lightweight JavaScript object that mirrors the structure of the real DOM. It is not magic, it is not secret, it is just an optimization so React can figure out what to change without touching the slow real DOM until it has to.

⬡ R3F Connection

React Three Fiber uses this exact same reconciler, but instead of syncing your description to the HTML DOM, it syncs it to a Three.js scene graph — a tree of 3D objects. The idea is identical: you describe the 3D scene, and R3F makes the world match your description. Every concept in this chapter transfers directly.

1.5 Chapter Recap

React exists because keeping UI in sync with data is hard, and React makes it easy by letting you describe the UI rather than manipulate it. The declarative idea — describe what, not how — is the foundation of everything else in this book.

Chapter 2

Thinking Declaratively

This is the chapter that makes or breaks everything that follows. Read it slowly.

2.1 Imperative: Step by Step

Imperative code tells the computer how to do something, one step at a time. If I asked you to make a sandwich imperatively, I would say: open the bread bag, remove two slices, place them on the counter, open the jar, take a knife, dip it, spread it, and so on. Every step is explicit.

Plain DOM code is imperative. Here is how you would toggle a class on a button when it is clicked:

const btn = document.querySelector('.btn');
btn.addEventListener('click', () => {
  btn.classList.toggle('active');
});

You find the button, you attach a listener, you toggle the class. Every action is your responsibility. If you have ten buttons, you do this ten times.

2.2 Declarative: Describe the Destination

Declarative code tells the computer what you want the result to look like and lets the computer figure out how to get there. If I asked you to make a sandwich declaratively, I would say: I want a peanut butter and jelly sandwich. Done.

Here is the same button in React:

import { useState } from 'react';

function Button() {
  const [active, setActive] = useState(false);
  return (
    <button
      className={active ? 'btn active' : 'btn'}
      onClick={() => setActive(!active)}
    >Click me</button>
  );
}

You are not finding the button. You are not toggling a class. You are describing the button's appearance as a function of its state. React does the rest.

🔑 Key Idea

Imperative code says how. Declarative code says what. React components are what.

2.3 Declarations Change with Data

A common confusion: "if I just describe the UI once, how does it change?" The answer is that your description is a function of your data. When the data changes, React re-runs the description and updates the DOM to match. The description itself is static; its output changes because its inputs change.

🧠 Mental Model

Imperative code is like walking a friend through a city by giving them turn-by-turn directions. Declarative code is like handing them a map and saying "meet me at this café." The café does not move; the map does not change; but your friend can arrive from anywhere.

2.4 Why This Is Harder at First, Easier Later

Declarative thinking feels strange at first because you lose the sense of control. You are not pulling levers; you are describing a destination. But once the mindset clicks, something beautiful happens: your code stops fighting itself. You no longer chase "where does this class get added?" bugs. The description is the truth, and the truth is always up to date.

⬡ R3F Connection

Three.js in its raw form is extremely imperative. You create a scene, create a mesh, set its position, add it to the scene, remember to remove it later, manage disposal, and so on. React Three Fiber takes all of that and makes it declarative. You describe your 3D scene with JSX, and R3F handles the rest. Everything you learn about declarative thinking here transfers directly and saves you massive amounts of boilerplate in 3D.

2.5 Chapter Recap

Imperative code tells the computer how to work; declarative code describes the result. React is declarative. Your UI is a description that changes as your data changes. In three dimensions with R3F, the same idea applies to 3D scenes.

Chapter 3

Your First Component and JSX

Components are the atoms of every React app. Nothing else makes sense without them.

3.1 What Is a Component?

A React component is a function that returns a description of some UI. That is the entire definition. When React runs your component, it gets back a description — a tree of elements — and it knows how to turn that tree into real DOM nodes on the screen.

Components are reusable. If you define a PropertyCard component once, you can use it a hundred times in your app, each with different data. Components compose: a component can use other components inside itself, and those can use other components, and before long you have a tree — a component tree — that describes your whole application.

function Welcome() {
  return <h1>Hello, React!</h1>;
}

That is a real, complete, valid React component. It is a function. It returns a description of an h1 element. That is all.

3.2 JSX: HTML in Your JavaScript

That strange-looking tag in the return statement is JSX. JSX is a syntax extension that lets you write what looks like HTML inside JavaScript. Under the hood, a tool called a bundler (like Vite) transforms JSX into regular JavaScript function calls. You do not need to think about the transformation — just know that JSX is a convenient way to describe a tree of elements.

🔑 Key Idea

JSX is not HTML. It looks like HTML so it feels familiar, but it is JavaScript. Every JSX tag becomes a function call that creates a description of an element.

3.3 The Rules of JSX

JSX has a few rules that will trip you up once if you do not see them now:

  1. A component must return a single root element. If you want to return siblings, wrap them in a fragment: <>...</>.
  2. HTML's class attribute becomes className in JSX, because class is a reserved word in JavaScript.
  3. All tags must be closed. Void elements like img and br become <img /> and <br />.
  4. JavaScript expressions inside JSX go in curly braces: <h1>{title}</h1>.
  5. Component names must start with a capital letter. Lowercase names are treated as HTML tags.
⚠ Common Mistake

Writing function propertyCard() with a lowercase name. React will silently treat your component as an HTML tag and not render it. Always capitalize component function names.

3.4 Nesting Components: The Tree Begins

Once you have components, you can use them inside other components just like HTML tags:

function PropertyCard() {
  return <div>A property</div>;
}

function App() {
  return (
    <div>
      <h1>My Portfolio</h1>
      <PropertyCard />
      <PropertyCard />
    </div>
  );
}

The component tree here is: App contains a div, a heading, and two PropertyCards. Learning to see your UI as a tree is one of the most important skills in React.

Every React app is a tree of components. You are the architect of that tree.
🧠 Mental Model

Components are LEGO bricks. Small, self-contained, reusable. You snap them together to build bigger things. The component tree is what you end up with when you are done building.

⬡ R3F Connection

In R3F, your 3D scene is also a component tree. The root is the Canvas component. Inside it, you nest meshes, groups, lights, and cameras — all of them components. The tree you see in this chapter is the same idea you will see in a 3D scene in Book 2. The only difference is the names of the tags.

3.5 Chapter Recap

A React component is a function that returns a JSX description of some UI. JSX looks like HTML but is really JavaScript. Components can nest inside each other to form a tree. The component tree is the structural backbone of every React app — and every R3F scene.

✏ Exercise 3B — Practice

Identify and fix the five mistakes in this JSX:

function broken() {
  return
    <h1 class="title">Hi</h1>
    <p>Paragraph
}

Five problems:

  1. Lowercase component name — broken must be Broken (or any capitalized name). Lowercase names are treated as HTML tags.
  2. return on its own line — JavaScript's Automatic Semicolon Insertion (ASI) silently inserts a semicolon after return, making the function return undefined and the JSX below it dead code. Always wrap multi-line JSX in parentheses: return ( ... ).
  3. class instead of className — class is a reserved word in JavaScript. JSX uses className.
  4. Two sibling elements without a wrapper — JSX must return a single root element. Wrap siblings in a <div> or a fragment <>...</>.
  5. Unclosed <p> tag — All JSX tags must be explicitly closed.
function Fixed() {
  return (
    <div>
      <h1 className="title">Hi</h1>
      <p>Paragraph</p>
    </div>
  );
}

The bare-return / ASI trap is problem #2 above and the one most beginners miss. You will hit it the first time you write a component with multi-line JSX — now you know why and how to fix it.

Chapter 4

Props — Giving Components Their Instructions

Components are functions. Props are their arguments.

4.1 The Problem Props Solve

In the last chapter you wrote a PropertyCard that rendered "Modern Loft." If you needed five different cards with five different titles, you would have to write five different components. That is obviously silly. Props are the solution: they let a single component behave differently based on the data passed into it.

4.2 Passing Props from the Caller

<PropertyCard title="Modern Loft" price={450000} />
<PropertyCard title="Coastal Villa" price={1200000} />

Each instance gets different data. To the caller, props look just like HTML attributes.

4.3 Receiving Props Inside the Component

Inside the component function, props arrive as the first argument — an object with one property per prop.

function PropertyCard({ title, price }) {
  return (
    <div>
      <h2>{title}</h2>
      <p>${price.toLocaleString()}</p>
    </div>
  );
}

The curly braces after the function name are destructuring — a JavaScript feature that pulls the title and price fields out of the props object. You could also write function PropertyCard(props) and then use props.title and props.price, but destructuring is cleaner and is what you will see in all modern React code.

4.4 Props Are Read-Only

You must not modify a prop inside your component. Props are a one-way message from parent to child, and modifying them would break the predictable data flow that makes React easy to reason about.

🔑 Key Idea

Props flow down, never sideways, never up. A component receives props from its parent and uses them to decide what to render. It does not change them.

4.5 The Children Prop

There is one special prop: children. It is whatever you put between the opening and closing tags of a component. It is how you build wrappers and layouts.

function Card({ children }) {
  return <div className="card">{children}</div>;
}

// Usage:
<Card>
  <h2>Anything in here</h2>
</Card>
🧠 Mental Model

A component is a function; props are its arguments; the return value is a description of the UI for those arguments. Same component, different props, different outputs.

4.6 TypeScript Tip — Typing Props with an Interface

This is the first place TypeScript really starts to help. You can declare exactly what props a component expects:

interface PropertyCardProps {
  title: string;
  price: number;
  location?: string; // optional
}

function PropertyCard({ title, price, location }: PropertyCardProps) {
  return (
    <div>
      <h2>{title}</h2>
      <p>{location ?? 'Unknown location'}</p>
      <p>${price.toLocaleString()}</p>
    </div>
  );
}
TS TypeScript Tip

The ? after a field name means the prop is optional. TypeScript will now catch any call to PropertyCard that forgets title or passes the wrong type, before your code even runs.

Chapter 5

State — When Components Need to Remember

Props come from outside. State lives inside. That distinction is the whole chapter.

5.1 Why State Exists

Props let a component receive data from its parent, but some data does not come from the parent. It belongs to the component itself. Whether a card is expanded, whether a button is pressed, whether an input holds the value the user just typed — these are internal to the component. State is React's mechanism for holding this internal data and reacting when it changes.

5.2 Introducing useState

import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  );
}

useState is a hook — a special function React provides for giving components powers they would not otherwise have. It returns a pair: the current state value and a function to update it. You destructure them into local variables, conventionally named [thing, setThing].

5.3 What Happens When You Call setCount

Calling setCount(count + 1) schedules an update and signals React to re-render. It does not immediately change count. In the current render, count still holds its old value — the new value only appears in the next render, when React re-runs your component function from the top.

This is why the following does not increment by 2:

// Both calls read the same stale `count` from the current render
setCount(count + 1);
setCount(count + 1); // still count + 1, not count + 2

When your next state depends on the current value, use the functional update form — pass a function instead of a value. React will call your function with the guaranteed latest state:

// Correct: each call receives the most recent value
setCount(prev => prev + 1);
setCount(prev => prev + 1); // correctly increments by 2
🔑 Key Idea

Setting state is not assignment — it is a request to React to re-render with a new value. count does not change in the current render. When the new value depends on the old one, use setCount(prev => prev + 1) to guarantee you are reading the latest state.

5.4 State vs Props

PropsState
Come from the parentOwned by the component
Read-only inside the componentUpdatable via setter
Same for every render unless parent changes themPersists across renders; changes when setter is called
Think of them as function argumentsThink of them as instance memory

5.5 State Is Private Per Instance

If you render five Counter components, each one has its own independent count. State is local to the component instance — not shared between instances. This is how you can have ten PropertyCards on a page, each with its own favorited state, without any of them interfering with the others.

5.6 Lifting State Up

Sometimes two sibling components need to share the same piece of state. For example, a filter bar and a property grid both need to know the current search term. The solution is to move the state up to their nearest common parent and pass it down as props. This is called "lifting state up" and it is a recurring pattern.

// State lives in the parent — both children read from it
function App() {
  const [search, setSearch] = useState('');
  return (
    <>
      // FilterBar writes the shared state via the setter passed as a prop
      <FilterBar search={search} onSearchChange={setSearch} />
      // PropertyGrid reads the shared state via a prop
      <PropertyGrid search={search} />
    </>
  );
}

function FilterBar({ search, onSearchChange }) {
  return (
    <input
      value={search}
      onChange={(e) => onSearchChange(e.target.value)}
      placeholder="Search residences..."
    />
  );
}

function PropertyGrid({ search }) {
  // Filters its list based on the search term from the parent
  return <p>Showing results for: {search}</p>;
}

The state owns search. FilterBar gets a setter passed as a prop and calls it on change. PropertyGrid receives the current value as a read-only prop. Neither sibling talks to the other — they both talk to the parent. This is the pattern you will use every time two components need to stay in sync.

🧠 Mental Model

State is a component's personal notebook. Nobody else can read it. When the component writes a new page, it asks React to redraw what it shows the world.

⚠ Common Mistake

Mutating state directly instead of calling the setter. count++ or list.push(x) will not cause React to re-render. You must call setCount or setList([...list, x]). Always use the setter; always produce a new value rather than mutating the old one.

🔑 Key Idea — Functional Updates

When the new state depends on the previous one, pass a function to the setter instead of a value: setCount(prev => prev + 1). This guarantees you are reading the latest state, even if multiple updates are batched together by React. Use this form any time you write setSomething(something + 1) or setSomething(!something).

5.7 TypeScript Tip — Typing State

Most of the time, TypeScript can infer the type of state from the initial value:

const [count, setCount] = useState(0); // inferred as number
const [name, setName] = useState(''); // inferred as string

When the initial value does not fully describe the type — for example, state that might be a string or null — you can make it explicit:

const [user, setUser] = useState<string | null>(null);

Chapter 7

Understanding Re-rendering

Most React courses bury this chapter. For R3F, it comes now.

7.1 The Render Cycle in Two Phases

When a state setter is called, React goes through two phases. First, the render phase: React calls your component function, you return a new JSX description of the UI, and React compares the new description to the previous one (this comparison step is called reconciliation). Second, the commit phase: React applies only the differences to the real DOM.

The render phase is fast because it is just running JavaScript functions and comparing objects. The commit phase is slower because it touches the real DOM. React's design keeps as much work as possible in the fast phase, and minimizes the slow one.

🔑 Key Idea

React's official model has two phases: render (run functions, diff the result) and commit (update the DOM). You will see "reconciliation" mentioned in documentation — it refers to the diffing step inside the render phase, not a separate phase.

7.2 What Triggers a Re-render

A component re-renders when:

  • Its own state changes (you called a setter).
  • Its props change (the parent passed different values).
  • Its parent re-renders (by default, children re-render too).

This last point is the one that surprises beginners. If a parent re-renders, all its children re-render as well, even if their props did not change. React then compares each child's new output to the old one and usually finds they are identical, so nothing actually hits the DOM. The render function runs, but the DOM does not update.

🔑 Key Idea

"Re-render" does not mean "the DOM updated." It means "React called the component function again." Whether the DOM changes depends on whether the new output differs from the old.

7.3 Why This Matters for 3D

A 3D scene runs at 60 frames per second. If you accidentally cause a re-render every frame — for example, by driving an animation with state — you will bring your scene to a crawl. In R3F, you learn to use useRef and the useFrame hook instead of state for animation. That is a Book 2 topic, but the intuition starts here: understand what triggers a re-render so you can avoid the ones you do not want.

🧠 Mental Model

Think of React as a photographer. Every render is a new photograph of your UI. The reconciler compares the new photo to the last one and only updates what changed. The commit is the part where the developed photo gets pinned to the wall.

⬡ R3F Connection

R3F has two separate loops: React's render cycle (for structural changes) and the animation loop via useFrame (for per-frame updates). Understanding that they are different is one of the most important realizations for R3F beginners. Every frame of animation should NOT trigger a React re-render — that would be disastrous for performance.

Lists and Keys

Every real app renders arrays of data. Do it carelessly and strange bugs appear.

9.1 Rendering a List with .map()

To render a collection, you turn an array of data into an array of elements using JavaScript's .map():

interface Property {
  id: string;
  title: string;
  price: number;
}

function PropertyList({ properties }: { properties: Property[] }) {
  return (
    <div>
      {properties.map(p => (
        <PropertyCard key={p.id} title={p.title} price={p.price} />
      ))}
    </div>
  );
}

9.2 The key Prop

Every element in a list needs a unique key prop. React uses keys to match items between renders — to know which items are new, which were removed, and which just moved. Without keys, React may reuse DOM nodes for the wrong items and your UI will develop subtle bugs.

🔑 Key Idea

A good key is stable, unique, and comes from the data itself — usually an id field. Avoid using the array index as a key if the list can be reordered or filtered.

⚠ Common Mistake

Using index as a key in a list that can be sorted, filtered, or have items inserted at the start. The index belongs to the position in the array, not to the item, so React ends up matching the wrong items and strange state bugs appear.

⬡ The Book Two Layer

In the immersive version of Aurelia, you will map over the same residences array to render one 3D scene per residence. The key pattern applies identically. Good keys mean React Three Fiber can efficiently add, swap, and remove 3D residences as the data changes.

useRef — The Imperative Escape Hatch

Sometimes you need to reach through React and touch something directly. useRef is how.

11.1 Why useRef Exists

React prefers you describe things declaratively. But some things cannot be fully described: focusing an input, measuring a DOM element's size, holding a value that should survive re-renders but not trigger them. For these, React gives you useRef — a controlled hole through the abstraction.

import { useRef } from 'react';

// A ref lets us reach through React and touch the real DOM element
function FocusOnDemand() {
  const inputRef = useRef<HTMLInputElement>(null);

  return (
    <>
      <input ref={inputRef} placeholder="Type here..." />
      <button onClick={() => inputRef.current?.focus()}>
        Focus the input
      </button>
    </>
  );
}

The ref prop on the input tells React: "give me direct access to this DOM node." After the component renders, inputRef.current holds the actual <input> element. We can call any native DOM method on it — here, .focus() — from inside a button's click handler.

🔑 Key Idea

Attach the ref with the ref prop. After render, ref.current holds the real DOM node. Call native DOM methods on it from event handlers or — once you have learned useEffect in Ch 12 — from effects.

11.2 The Ref Does Not Trigger a Re-render

This is the point of refs. You can update ref.current freely and React will not re-render the component. For values that should persist across renders but whose changes should not redraw the UI, refs are the right tool.

🔑 Key Idea

A ref is a box that holds something. The box is stable across renders. What you put in the box can change without causing a re-render. This is exactly the opposite of state.

🧠 Mental Model

State is for data the user sees. Refs are for data the code needs to remember but the user does not directly see. If changing a value should redraw the screen, use state. If it should not, use a ref.

⬡ R3F Connection

This chapter is one of the most directly critical for R3F. In R3F, you attach a ref to a mesh and use it to access the underlying Three.js object. Inside the useFrame hook, you then mutate the mesh's position and rotation directly — at 60 frames per second — without ever touching React state. If you used state for animation, you would trigger sixty re-renders per second and crash the browser. useRef is the reason this works. Please re-read this chapter before you open Book 2.

Chapter 12

useEffect — Living with the Outside World

React describes the UI. useEffect is how React talks to everything else.

12.1 What Is a Side Effect?

A side effect is anything your component does that reaches outside its own rendering. Fetching data, subscribing to events, setting a timer, loading a texture, attaching a scroll listener — all side effects. They should not run during rendering; they should run after the render is committed to the DOM. useEffect is how you schedule them.

useEffect(() => {
  document.title = `Selected: ${title}`;
}, [title]);

12.2 The Dependency Array

The second argument to useEffect is a dependency array. It tells React: "re-run this effect only when one of these values changes." An empty array means "run once after the first render." No array at all means "run after every render" — which is usually wrong.

⚠ Common Mistake

Forgetting the dependency array and causing infinite loops: your effect updates state, which triggers a re-render, which runs the effect, which updates state again. Always think carefully about what your effect depends on.

12.3 Cleanup

If your effect creates something that needs to be cleaned up — an event listener, a timer, a subscription — return a cleanup function. React will call it before re-running the effect and when the component unmounts.

useEffect(() => {
  const id = setInterval(tick, 1000);
  return () => clearInterval(id);
}, []);
⬡ R3F Connection

In R3F, useEffect is where you load textures, subscribe to keyboard events, initialize non-React Three.js systems, and — most importantly — clean up resources when a component unmounts. Three.js objects hold GPU memory, and forgetting to dispose them leaks memory. The cleanup pattern you learn here is how you keep R3F scenes well-behaved.

Chapter 13

useMemo — Making Expensive Work Cheaper

Do the work once. Remember the answer. Skip the work next time if the inputs are the same.

13.1 The Problem

Every time a component re-renders, everything inside the function body runs again. Usually that is fine. But if you have an expensive calculation — sorting, filtering, computing a complex geometry — running it on every render is wasteful.

13.2 useMemo

const filtered = useMemo(
  () => properties.filter(p => p.price < maxPrice),
  [properties, maxPrice]
);

React will run the function the first time, remember the result, and skip running it on subsequent renders as long as the dependencies — properties and maxPrice — have not changed.

🔑 Key Idea

useMemo is a cache with inputs. If the inputs are the same as last time, return the cached result. If they changed, recompute.

⚠ Common Mistake

Memoizing everything. useMemo has a cost — React must track dependencies and compare them. For trivial calculations, it is slower than just recomputing. Memoize expensive work, not cheap work.

⬡ R3F Connection

Here is a real R3F component — a rotating box that changes color on hover. Read it slowly. Point at each piece and name what it is doing in Book 1 terms.

⚠ Before You Read

This component cannot run on its own. It needs two things provided by Book 2: (1) a <Canvas> wrapper — the R3F equivalent of mounting React to the DOM, and (2) at least one light in the scene. meshBasicMaterial below works without lights; in Book 2 you will switch to meshStandardMaterial once you know how to add lights. The React concepts are all ones you already know — the only new pieces are useFrame and the R3F-specific JSX tags.

import { useRef, useState } from 'react';
import { useFrame } from '@react-three/fiber';
import * as THREE from 'three';

// Must live inside a <Canvas> — Book 2 covers this setup
function SpinningBox() {
  // useRef — Ch 11: holds the mesh without triggering re-renders
  const meshRef = useRef<THREE.Mesh>(null);
  // useState — Ch 5: hover state drives color
  const [hovered, setHovered] = useState(false);

  // useFrame: NEW in R3F — runs every animation frame (60×/sec)
  // We mutate the mesh directly via the ref — no state, no re-renders
  useFrame((_, delta) => {
    if (meshRef.current) {
      meshRef.current.rotation.y += delta;
    }
  });

  // Composition — Ch 10: geometry and material are children of mesh
  return (
    <mesh
      ref={meshRef}
      // Events — Ch 6: same pattern as onClick on a button
      onPointerOver={() => setHovered(true)}
      onPointerOut={() => setHovered(false)}
    >
      <boxGeometry args={[1, 1, 1]} />
      // meshBasicMaterial works without lights — good for previews
      // Conditional rendering — Ch 8: ternary drives color from state
      <meshBasicMaterial color={hovered ? 'tomato' : 'royalblue'} />
    </mesh>
  );
}

Look at everything you recognize. useRef holds the mesh — Ch 11. useState holds the hover boolean — Ch 5. The return is JSX — Ch 3. The mesh has pointer event handlers, exactly like buttons had onClick — Ch 6. Its color is a ternary on state — Ch 8. The geometry and material are nested children — composition from Ch 10. The only truly new concept is useFrame, which runs every animation frame and directly mutates the mesh's rotation through the ref. You already know why this works: refs do not trigger re-renders, so mutating them sixty times per second is fast and safe — Ch 11.

W7 · Re-rendering — The Loop That Drives the Site

When state changes, React calls your component function again. The whole function. Top to bottom. With the new state value. It then compares the JSX you returned to the previous JSX and updates only the DOM nodes that differ.

This means three things, all important:

  • Local variables inside your function reset every render. They are not memory.
  • Anything you want to survive between renders has to live in useState, useRef, or props.
  • Re-renders are usually cheap. Do not pre-optimize a launch site you have not measured.

The Debugging Mindset

Bugs are not failures. They are clues.

Read the Error Message Twice

React error messages are unusually helpful. They tell you what went wrong, where, and often why. Do not skim them. Read them twice. Most of the time, the answer is in the first line.

The Three Questions

  1. What did I expect to happen? Be specific. Not "the button should work" but "clicking the button should increment the count by 1."
  2. What actually happened? Describe the observation, not the interpretation.
  3. What is the gap? The gap is where the bug lives.

Console.log Is Your Friend

Put console.log statements everywhere you are not sure. Log props. Log state. Log whether an effect is running. Remove them when you are done. This is not cheating — it is how professionals debug.

Check Your Renders

A surprising number of bugs come from the component not re-rendering when you expected it to, or re-rendering when you did not. Add a console.log('rendering X') at the top of a component to see what is actually running.

The Rubber Duck

When you are stuck, explain your problem, out loud, to an inanimate object — a rubber duck, a coffee mug, your cat. You will often solve the bug while describing it. This is a real technique and it works.
